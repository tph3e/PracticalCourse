\subsection{Resource Modeling}\label{subsec:resources}
\textit{(Aldo Patrone)}\quad
At every activity start the simulation must decide \emph{which} resources may
execute the activity (permissions), \emph{when} they are available
(availabilities) and \emph{which} free resource is chosen (allocation). These
three concerns are encapsulated behind a single \emph{ResourceEngine} facade:
it assigns a resource that is permitted, available and free, records it
on the event and releases it on activity completion. The selection step is a
swappable \emph{AllocationStrategy}, which is the integration point for the
optimized policies in Part~II.

\subsubsection{Resource Availabilities}
The basic model applies a fixed working interval (Mon--Fri, 08:00--18:00) to
every resource. The advanced model instead learns a \emph{per-resource calendar}:
for each resource it keeps the (weekday, hour) buckets in which that resource is
actually observed working, using a relative frequency threshold to suppress
off-hours noise. Writing $n_r(w,h)$ for resource $r$'s event count in bucket $(w,h)$,
\[
\mathrm{Cal}_r=\big\{(w,h): n_r(w,h)\ge \max(1,\ \tau\cdot\!\max_{w',h'} n_r(w',h'))\big\},
\]
$\tau=0.1$
and at time $t$ the available set is $A(t)=\{r:(\mathrm{wd}(t),\mathrm{hr}(t))\in\mathrm{Cal}_r\}$.
The design decision here is data-driven: BPIC-17 carries no
system-vs-human attribute and most resources act across several lanes, so a
fixed ``system always available'' rule is not supported by the data, whereas a
learned calendar per resource is. On the full log the learned calendars
(Fig.~\ref{fig:availability}) cover $96.4\%$ of events against $72.0\%$ for the fixed
interval (Table~\ref{tab:resource_eval}). This in-sample gap is, however, largely
optimistic: under a temporal holdout (calendars learned on the first $70\%$ of the log
period, coverage measured on the last $30\%$) the learned model reaches $73.1\%$ versus
$72.3\%$ for the fixed interval, only marginally more faithful out-of-sample, because
per-resource calendars are data-hungry. Their decisive advantage is instead
\emph{efficiency}: the learned model assumes only $44.4$ available hours per resource per
week against $50$ for the blanket Mon--Fri 08--18 window, concentrating availability on the
hours a resource actually works (Fig.~\ref{fig:avail_capacity}) at no loss of out-of-sample
faithfulness. The per-resource calendar breadth is shown in the Appendix
(Fig.~\ref{fig:cal_sizes}).

\emph{Limitations.} Availability is checked only at the allocation instant, so an activity
started just before a resource's window closes still runs to completion. Buckets use UTC hours
(the BPIC-17 convention), which shifts the absolute window by the Dutch offset but preserves the
relative weekday/hour structure. Finally, as the holdout shows, the calendars need sufficient history,
so sparsely observed resources receive narrow calendars.

\subsubsection{Resource Permissions}
The basic model is a resource-activity matrix: a resource may perform an
activity iff it has performed it before. The advanced model uses role
discovery with OrdinoR~\cite{yang2022ordinor}: resources are grouped into roles
and a resource may perform an activity iff its role's execution context covers
it. An observed-performer \emph{floor} is added so that every activity keeps at
least its historical executors, which guarantees that no activity becomes
unexecutable (avoiding resource deadlocks during simulation). On BPIC-17, $14$
discovered roles cover all $26$ activities and the role generalization adds
$872$ \mbox{(activity, resource)} permissions over the plain matrix, modeling
that resources within a role are interchangeable. The role count ($14$) matches the
organizational analysis from our first assignment. A sensitivity check with the in-repo
clustering fallback confirms the choice is not brittle (varying it over $8$--$20$ moves the
added permissions only modestly). We deliberately report the generalization critically: it is
permissive. On the committed model $20$ of $26$ activities already permit $\geq 140$ of
$149$ resources (matrix $85\%$ dense), so for most activities the permission constraint is
nearly non-binding and allocation is governed mainly by availability. A temporal split (train
$70\%$ / test $30\%$) shows it is only weakly predictive of unseen assignments: held-out
(activity, resource) recall rises from $77.4\%$ for the plain matrix to just $77.8\%$ with
roles. The added permissions therefore model within-role interchangeability rather than
validated future assignments, and it is the observed-performer \emph{floor}, not the role
expansion, that guarantees deadlock-free coverage. The per-activity permission breadth is shown
in the Appendix (Fig.~\ref{fig:perm_breadth}).

\subsubsection{Resource Allocation}
Allocation selects uniformly at random among the permitted, available and free
resources. It is the deliberately unbiased Part~I baseline: it lets the other
simulation components be evaluated without an allocation policy as a confounder.
Replacing this strategy with an optimized policy is the subject of Part~II and
requires no change to the availability, permission or core components. Its
uniformity over the candidate set is verified both visually (Appendix,
Fig.~\ref{fig:alloc_unif}) and statistically: a chi-square goodness-of-fit test on
$40{,}000$ draws, 
\[
\chi^2=\sum_{i=1}^{k}(O_i-E_i)^2/E_i 
\]
with $E_i=N/k$, does not reject
uniformity ($\chi^2=3.01$, $p=0.39$).
